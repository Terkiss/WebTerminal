## 2. 제품 개요

WebPowerShell은 사용자가 웹 브라우저에서 인증 후 여러 개의 PowerShell 세션을 탭 단위로 생성하고, 명령어를 실행하며, 실행 결과를 실시간으로 확인할 수 있도록 제공하는 관리용 웹 애플리케이션이다.

각 터미널 탭은 독립된 PowerShell Runspace를 사용하며, 탭 간 실행 환경과 출력 내용을 서로 격리한다.

모든 명령어 실행 기록은 감사 로그로 저장되며, 비밀번호 주기 변경, 세션 자동 종료, 로그인 시도 제한, 최소 권한 실행 등의 보안 정책을 적용한다.

---

## 3. 제품 목표

### 3.1 핵심 목표

1. 별도의 원격 접속 프로그램 없이 웹 브라우저에서 PowerShell 명령을 실행할 수 있어야 한다.
2. 한 사용자가 여러 PowerShell 세션을 동시에 사용할 수 있어야 한다.
3. 각 탭의 작업 상태와 출력 기록이 탭 전환 이후에도 유지되어야 한다.
4. 사용자가 실행한 모든 명령어를 감사 가능한 형태로 기록해야 한다.
5. 장시간 방치된 PowerShell 세션을 자동 종료해 서버 자원 누수를 방지해야 한다.
6. 관리자 권한 노출 및 임의 명령 실행으로 발생할 수 있는 보안 위험을 최소화해야 한다.

### 3.2 성공 기준

* 정상 명령어 실행 요청의 99% 이상이 오류 없이 처리된다.
* 명령 실행 후 최초 출력이 일반적인 서버 환경에서 2초 이내에 표시된다.
* 탭 전환 및 복귀 시 기존 터미널 출력이 손실되지 않는다.
* 종료된 탭이나 연결이 끊긴 세션의 Runspace가 정상적으로 해제된다.
* 모든 명령 실행에 대해 사용자, 세션, 명령어, 실행 시간, IP 주소가 기록된다.
* 비밀번호 만료 사용자는 허용된 API를 제외한 시스템 기능을 사용할 수 없다.
* 인증되지 않은 사용자는 PowerShell 세션을 생성하거나 명령을 실행할 수 없다.

## 4. 배경 및 문제 정의

기존 PowerShell 기반 서버 관리 작업은 원격 데스크톱, SSH, WinRM 또는 서버 직접 접속에 의존하는 경우가 많다.

이 방식에는 다음과 같은 문제가 있다.

* 사용자별 접근 통제가 어렵다.
* 실행 명령에 대한 중앙 감사 기록이 부족하다.
* 여러 작업을 동시에 수행할 때 세션 관리가 불편하다.
* 서버 접근 권한이 필요 이상으로 노출될 수 있다.
* 장시간 유지되는 원격 세션이 서버 자원을 점유할 수 있다.
* 별도의 클라이언트 설치나 네트워크 설정이 필요할 수 있다.

WebPowerShell은 브라우저 기반의 통합 접속 환경을 제공하여 이러한 문제를 해결한다.

## 5. 대상 사용자

### 5.1 일반 운영 사용자

서버 상태 확인, 로그 조회, 서비스 점검 등 사전에 허용된 PowerShell 명령을 실행하는 사용자다.

주요 요구사항은 다음과 같다.

* 간편한 로그인
* 여러 작업을 위한 다중 탭
* 실시간 명령 출력
* 기존 출력 내용 유지
* 세션 종료 기능
* 비밀번호 변경 안내

### 5.2 시스템 관리자

사용자 계정, 접근 권한, 감사 로그 및 서비스 운영 상태를 관리하는 사용자다.

초기 MVP에서는 사용자 계정을 데이터베이스 또는 별도 관리 도구를 통해 사전 발급할 수 있다. 관리자용 웹 화면은 후속 버전에서 제공한다.

### 5.3 보안 감사 담당자

사용자가 언제, 어디서, 어떤 명령을 실행했는지 확인하는 사용자다.

감사 담당자는 감사 로그 조회 권한을 가져야 하며 PowerShell 명령 실행 권한과는 분리되어야 한다.

## 7. 핵심 제품 원칙

### 7.1 세션 격리

각 터미널 탭은 다른 탭과 독립된 Runspace를 사용해야 한다.

한 탭에서 변경한 현재 디렉터리, 변수, 함수 또는 모듈 상태는 다른 탭에 영향을 주지 않아야 한다.

### 7.2 최소 권한

PowerShell 명령은 WebPowerShell 애플리케이션 프로세스의 운영체제 권한으로 실행된다.

애플리케이션은 Administrator, LocalSystem 또는 이와 동등한 고권한 계정으로 실행해서는 안 된다.

### 7.3 감사 가능성

사용자가 실행한 명령어는 누락 없이 기록되어야 한다.

감사 로그는 일반 사용자가 수정하거나 삭제할 수 없어야 한다.

### 7.4 자원 회수

닫힌 탭, 연결이 끊긴 브라우저 및 유휴 세션의 Runspace와 PowerShell 객체는 반드시 해제되어야 한다.

### 7.5 실패 안전성

인증 상태, 세션 소유권 또는 권한 검증이 불가능한 경우 명령 실행을 허용하지 않아야 한다.

## 8. 사용자 흐름

### 8.1 로그인 흐름

1. 사용자가 로그인 화면에 접속한다.
2. 사용자가 아이디와 비밀번호를 입력한다.
3. 서버가 계정 존재 여부, 활성 상태 및 비밀번호를 검증한다.
4. 인증에 성공하면 HttpOnly 및 Secure 속성이 적용된 인증 쿠키를 발급한다.
5. 비밀번호 변경 후 7일이 경과한 경우 비밀번호 변경 화면으로 이동한다.
6. 비밀번호가 유효한 경우 터미널 메인 화면으로 이동한다.

### 8.2 비밀번호 만료 흐름

1. 인증된 사용자가 API 또는 SignalR 기능을 요청한다.
2. 서버가 마지막 비밀번호 변경 시간을 확인한다.
3. 변경 후 168시간 이상 경과한 경우 요청을 차단한다.
4. 비밀번호 변경 및 로그아웃 기능만 허용한다.
5. 사용자가 기존 비밀번호와 새 비밀번호를 입력한다.
6. 변경에 성공하면 비밀번호 해시와 마지막 변경 시간을 갱신한다.
7. 사용자는 터미널 기능을 다시 사용할 수 있다.

### 8.3 새 탭 생성 흐름

1. 사용자가 터미널 화면의 `+` 버튼을 선택한다.
2. 프론트엔드가 새로운 고유 Tab ID를 생성한다.
3. xterm.js 터미널 인스턴스와 전용 DOM 컨테이너를 생성한다.
4. SignalR의 `OpenTab` 메서드를 호출한다.
5. 서버가 해당 사용자 연결과 Tab ID에 대응하는 Runspace를 생성한다.
6. 생성 성공 후 PowerShell 프롬프트를 표시한다.

### 8.4 명령 실행 흐름

1. 사용자가 활성 탭에 명령어를 입력한다.
2. 사용자가 Enter 키를 누른다.
3. 프론트엔드가 현재 Tab ID와 명령어를 SignalR로 전송한다.
4. 서버가 인증 상태와 탭 소유권을 검증한다.
5. 서버가 감사 로그를 기록한다.
6. 해당 탭의 Runspace에서 명령어를 실행한다.
7. 표준 출력과 오류 출력을 실시간으로 클라이언트에 전송한다.
8. 프론트엔드는 Tab ID에 해당하는 xterm.js 인스턴스에 결과를 출력한다.
9. 명령 실행이 종료되면 새로운 프롬프트를 표시한다.

### 8.5 탭 전환 흐름

1. 사용자가 다른 탭을 선택한다.
2. 선택한 탭의 터미널 컨테이너를 표시한다.
3. 나머지 터미널 컨테이너는 숨긴다.
4. 숨겨진 탭에서도 서버 출력 수신과 터미널 버퍼 갱신은 계속된다.
5. 사용자가 기존 탭으로 돌아오면 이전 출력 내용을 그대로 확인할 수 있다.

### 8.6 탭 종료 흐름

1. 사용자가 탭의 닫기 버튼을 선택한다.
2. 실행 중인 명령이 있는 경우 종료 여부를 안내한다.
3. 프론트엔드가 서버에 탭 종료를 요청한다.
4. 서버가 해당 PowerShell 실행을 중단한다.
5. PowerShell 및 Runspace 객체를 Dispose 처리한다.
6. 서버 세션 저장소에서 탭 정보를 제거한다.
7. 프론트엔드는 터미널 인스턴스와 DOM 요소를 제거한다.

### 8.7 연결 종료 흐름

1. 브라우저 종료, 네트워크 장애 또는 로그아웃으로 SignalR 연결이 종료된다.
2. 서버는 연결 ID에 연결된 모든 PowerShell 세션을 조회한다.
3. 실행 중인 명령을 중단한다.
4. 모든 PowerShell 및 Runspace 객체를 해제한다.
5. 연결에 속한 세션 정보를 삭제한다.

## 9. 기능 요구사항

## 9.1 인증 및 계정

### FR-AUTH-001 로그인

시스템은 사용자 아이디와 비밀번호를 이용한 로그인을 제공해야 한다.

**수용 기준**

* 유효한 계정과 비밀번호 입력 시 인증 쿠키가 발급된다.
* 존재하지 않는 계정과 잘못된 비밀번호는 동일한 오류 메시지를 반환한다.
* 비활성 계정은 로그인할 수 없다.
* 오류 메시지로 계정 존재 여부를 추측할 수 없어야 한다.

### FR-AUTH-002 인증 쿠키

인증 정보는 HttpOnly 쿠키로 관리해야 한다.

**수용 기준**

* 쿠키에 `HttpOnly` 속성이 적용된다.
* 운영 환경에서 `Secure` 속성이 적용된다.
* 적절한 `SameSite` 정책이 적용된다.
* 인증 토큰을 Local Storage 또는 Session Storage에 저장하지 않는다.

### FR-AUTH-003 로그아웃

사용자는 현재 인증 세션을 종료할 수 있어야 한다.

**수용 기준**

* 로그아웃 시 인증 쿠키가 무효화된다.
* 해당 SignalR 연결의 모든 PowerShell 세션이 종료된다.
* 로그아웃 이후 보호된 API에 접근할 수 없다.

### FR-AUTH-004 비밀번호 만료

마지막 비밀번호 변경 후 168시간이 경과한 사용자는 비밀번호를 변경해야 한다.

**수용 기준**

* 만료 사용자는 비밀번호 변경과 로그아웃을 제외한 기능을 사용할 수 없다.
* 일반 API와 SignalR Hub 연결 모두 동일한 정책을 적용받는다.
* 만료 응답은 클라이언트가 식별 가능한 오류 코드 `PasswordExpired`를 포함한다.
* 비밀번호 변경 완료 후 만료 상태가 즉시 해제된다.

### FR-AUTH-005 비밀번호 변경

사용자는 기존 비밀번호를 검증한 후 새 비밀번호로 변경할 수 있어야 한다.

**수용 기준**

* 기존 비밀번호가 일치하지 않으면 변경되지 않는다.
* 새 비밀번호는 비밀번호 정책을 충족해야 한다.
* 비밀번호는 평문으로 저장되지 않는다.
* 변경 성공 시 `LastPasswordChangeDate`가 UTC 기준 현재 시각으로 갱신된다.
* 기존 비밀번호와 동일한 비밀번호 사용 허용 여부는 운영 정책으로 설정할 수 있다.

### FR-AUTH-006 로그인 시도 제한

로그인 API는 무차별 대입 공격을 제한해야 한다.

**수용 기준**

* IP 주소 또는 IP와 계정 조합을 기준으로 요청 횟수를 제한한다.
* 기본 정책은 1분 내 로그인 실패 5회로 한다.
* 제한 초과 시 일정 시간 동안 추가 로그인을 차단한다.
* Rate Limit 상태는 HTTP 429로 응답한다.
* 프록시 환경에서는 신뢰할 수 있는 Forwarded Header 설정을 적용한다.

## 9.2 터미널 탭 관리

### FR-TAB-001 탭 생성

사용자는 새로운 PowerShell 탭을 생성할 수 있어야 한다.

**수용 기준**

* 각 탭은 고유한 Tab ID를 가진다.
* 동일 연결에서 Tab ID가 중복될 수 없다.
* 탭마다 별도의 PowerShell Runspace가 생성된다.
* 세션 생성 실패 시 빈 탭을 유지하지 않고 오류를 표시한다.

### FR-TAB-002 탭 수 제한

시스템은 사용자 또는 연결별 최대 탭 수를 제한할 수 있어야 한다.

**기본값**

* 연결당 최대 5개
* 사용자당 최대 10개

**수용 기준**

* 제한 초과 시 새 탭이 생성되지 않는다.
* 사용자에게 최대 탭 수 초과 메시지를 표시한다.
* 최대 탭 수는 설정 파일 또는 환경 변수로 변경할 수 있다.

### FR-TAB-003 탭 이름

각 탭은 사용자에게 식별 가능한 이름을 표시해야 한다.

**수용 기준**

* 기본 이름은 `PowerShell 1`, `PowerShell 2` 형식으로 생성된다.
* 탭 이름 변경 기능은 선택 기능으로 제공할 수 있다.
* 탭 이름은 PowerShell 세션 식별자로 사용하지 않는다.

### FR-TAB-004 탭 전환

사용자는 실행 중인 여러 탭 사이를 전환할 수 있어야 한다.

**수용 기준**

* 탭 전환 시 기존 출력이 유지된다.
* 비활성 탭의 서버 출력도 해당 탭 버퍼에 계속 반영된다.
* 탭 전환은 PowerShell 세션 상태에 영향을 주지 않는다.

### FR-TAB-005 탭 닫기

사용자는 개별 터미널 탭을 닫을 수 있어야 한다.

**수용 기준**

* 탭 닫기 시 서버 Runspace가 해제된다.
* 실행 중인 명령이 있으면 중단 처리된다.
* 이미 종료된 탭에 대한 요청은 안전하게 무시하거나 명확한 오류를 반환한다.
* 다른 사용자의 탭을 종료할 수 없다.

---

## 9.3 PowerShell 명령 실행

### FR-PS-001 명령어 전송

사용자는 활성 터미널에서 PowerShell 명령어를 제출할 수 있어야 한다.

**수용 기준**

* Enter 입력 시 현재 입력 버퍼의 명령어가 서버로 전송된다.
* 빈 명령은 실행하지 않거나 새 프롬프트만 표시한다.
* 명령어 최대 길이를 제한한다.
* 기본 최대 길이는 16KB로 한다.
* 존재하지 않거나 종료된 세션에는 명령을 실행하지 않는다.

### FR-PS-002 세션 상태 유지

각 탭은 명령 실행 간 PowerShell 상태를 유지해야 한다.

**수용 기준**

* `Set-Location` 실행 후 다음 명령에서도 변경된 경로가 유지된다.
* 탭에서 생성한 변수와 함수가 동일 탭에서 유지된다.
* 다른 탭에는 상태가 공유되지 않는다.

### FR-PS-003 출력 스트리밍

명령 실행 결과는 완료 시점까지 기다리지 않고 가능한 범위에서 실시간으로 전달해야 한다.

**수용 기준**

* 표준 출력이 SignalR을 통해 해당 탭으로 전달된다.
* 출력 이벤트에는 Tab ID가 포함된다.
* 다른 탭이나 다른 사용자에게 출력이 전달되지 않는다.
* 대용량 출력에 대한 메시지 크기와 전송 속도 제한을 적용한다.

### FR-PS-004 오류 출력

PowerShell 오류는 일반 출력과 구분 가능한 형태로 전달해야 한다.

**수용 기준**

* PowerShell Error Stream을 별도로 수집한다.
* 오류 메시지는 터미널에서 시각적으로 구분할 수 있다.
* 서버 내부 스택 트레이스나 민감한 경로는 기본적으로 사용자에게 노출하지 않는다.
* 상세 오류는 서버 로그에 기록한다.

### FR-PS-005 명령 중복 실행 방지

동일한 탭에서 동시에 여러 명령이 Runspace에 실행되지 않도록 제어해야 한다.

**수용 기준**

* 탭별 명령 실행 잠금 또는 큐를 적용한다.
* 실행 중 추가 명령을 입력하는 경우 대기시키거나 `SessionBusy` 오류를 반환한다.
* 동일 PowerShell 객체에 대한 동시 `Commands.Clear()` 및 `InvokeAsync()` 호출이 발생하지 않는다.

### FR-PS-006 명령 중단

사용자는 실행 중인 명령을 중단할 수 있어야 한다.

**수용 기준**

* 터미널의 중단 키 또는 별도 중지 버튼을 통해 중단 요청을 보낼 수 있다.
* 서버는 해당 PowerShell 인스턴스에 중단 요청을 전달한다.
* 중단된 세션이 계속 사용 가능한 상태인지 검증한다.
* 세션 복구가 불가능하면 기존 Runspace를 폐기하고 새 세션을 생성한다.

### FR-PS-007 프롬프트 표시

명령 실행 준비 상태를 나타내는 프롬프트를 표시해야 한다.

**수용 기준**

* 세션 생성 후 최초 프롬프트가 표시된다.
* 명령 실행 완료 또는 실패 후 새로운 프롬프트가 표시된다.
* 가능한 경우 현재 작업 경로를 프롬프트에 반영한다.

### FR-PS-008 대화형 명령 제한

MVP는 한 줄 또는 완성된 스크립트 단위의 명령 실행을 기본으로 한다.

`Read-Host`, 자격 증명 입력, 대화형 프로그램, 전체 화면 콘솔 앱과 같이 지속적인 표준 입력을 요구하는 명령은 완전하게 지원하지 않을 수 있다.

**수용 기준**

* 지원되지 않는 대화형 명령의 제약사항을 사용자에게 안내한다.
* 명령이 입력 대기 상태로 무기한 유지되지 않도록 실행 시간 제한을 적용할 수 있다.
* 자격 증명이나 보안 문자열을 터미널 평문으로 입력하도록 유도하지 않는다.

## 9.4 세션 관리

### FR-SESSION-001 세션 식별

서버의 내부 세션 키는 SignalR 연결과 Tab ID를 조합해 고유하게 식별해야 한다.

**수용 기준**

* 다른 연결에서 동일한 Tab ID를 사용하더라도 세션이 충돌하지 않는다.
* 클라이언트가 전달한 세션 ID만으로 다른 사용자의 세션에 접근할 수 없다.
* 서버는 `Context.ConnectionId`와 인증 사용자 정보를 기준으로 소유권을 확인한다.

### FR-SESSION-002 유휴 시간 갱신

사용자 활동이 발생할 때 세션의 마지막 활동 시간을 갱신해야 한다.

**활동으로 간주하는 작업**

* 명령 실행 시작
* 출력 전송
* 명령 중단
* 명시적인 세션 유지 신호

### FR-SESSION-003 유휴 세션 자동 종료

30분 이상 활동이 없는 PowerShell 세션은 자동 종료해야 한다.

**수용 기준**

* 백그라운드 작업이 기본 5분 간격으로 세션을 검사한다.
* 30분 이상 유휴 상태인 세션을 저장소에서 원자적으로 제거한다.
* PowerShell과 Runspace를 모두 Dispose 처리한다.
* 이미 종료 중인 세션을 중복 해제해 예외가 발생하지 않는다.
* 종료된 탭에 명령을 보내면 `SessionExpired` 오류가 반환된다.

### FR-SESSION-004 연결 종료 정리

SignalR 연결이 종료되면 연결에 속한 모든 세션을 종료해야 한다.

**수용 기준**

* 정상 종료와 비정상 종료 모두 처리한다.
* 일부 세션 종료 실패가 나머지 세션 정리를 막지 않는다.
* 정리 결과와 오류가 서버 로그에 기록된다.

### FR-SESSION-005 애플리케이션 종료 정리

애플리케이션 종료 시 메모리에 남아 있는 모든 PowerShell 세션을 정리해야 한다.

**수용 기준**

* 호스트 종료 토큰을 수신하면 신규 명령 실행을 차단한다.
* 실행 중인 명령을 중단한다.
* 모든 PowerShell 및 Runspace 객체를 Dispose 처리한다.

## 9.5 감사 로그

### FR-AUDIT-001 명령 실행 로그

모든 명령 실행 시도는 감사 로그로 기록해야 한다.

**필수 기록 항목**

* 사용자 ID
* 사용자명 또는 사용자 식별 스냅샷
* Tab ID
* 서버 내부 세션 식별자
* 실행 명령어
* 실행 요청 시각
* 완료 시각
* 접속 IP 주소
* 실행 결과 상태
* 오류 발생 여부
* 실행 소요 시간

### FR-AUDIT-002 실패 명령 기록

문법 오류, 권한 오류, 실행 중단 및 서버 오류가 발생한 명령도 기록해야 한다.

### FR-AUDIT-003 감사 로그 무결성

일반 사용자는 감사 로그를 수정하거나 삭제할 수 없어야 한다.

**수용 기준**

* 감사 로그 작성 권한과 조회 권한을 분리할 수 있다.
* 애플리케이션 계정에 불필요한 감사 로그 삭제 권한을 부여하지 않는다.
* 감사 로그 보존 기간은 운영 정책으로 설정한다.

### FR-AUDIT-004 민감 정보 처리

명령어에는 비밀번호, 토큰 또는 개인정보가 포함될 수 있다.

**수용 기준**

* 알려진 민감 파라미터를 마스킹할 수 있는 정책을 지원한다.
* 비밀번호 변경 API의 비밀번호는 감사 로그에 저장하지 않는다.
* 명령어 원문 저장 여부는 조직의 보안 정책과 개인정보 정책에 따라 결정한다.
* 민감 정보 마스킹이 불가능한 경우 해당 위험을 운영 정책에 명시한다.

## 9.6 오류 처리

### FR-ERR-001 공통 오류 코드

클라이언트는 서버 오류를 코드 기반으로 처리할 수 있어야 한다.

**주요 오류 코드**

| 오류 코드               | 설명               |
| ------------------- | ---------------- |
| `Unauthorized`      | 인증되지 않은 사용자      |
| `Forbidden`         | 기능 접근 권한 없음      |
| `PasswordExpired`   | 비밀번호 변경 필요       |
| `SessionNotFound`   | 존재하지 않는 세션       |
| `SessionExpired`    | 유휴 시간 초과로 종료된 세션 |
| `SessionBusy`       | 세션에서 다른 명령 실행 중  |
| `TabLimitExceeded`  | 최대 탭 수 초과        |
| `CommandTooLong`    | 명령어 길이 제한 초과     |
| `CommandTimedOut`   | 명령 실행 시간 초과      |
| `RateLimitExceeded` | 요청 횟수 제한 초과      |
| `InternalError`     | 서버 내부 오류         |

### FR-ERR-002 사용자 메시지

사용자에게는 이해 가능한 오류 메시지를 제공하되 서버 내부 구현 정보는 노출하지 않아야 한다.

## 10. 비기능 요구사항

## 10.1 성능

### NFR-PERF-001 응답 시간

* 일반 API 요청의 95%는 1초 이내에 응답해야 한다.
* PowerShell 명령 실행 후 최초 출력은 일반적인 조건에서 2초 이내에 표시되어야 한다.
* 탭 전환은 200ms 이내에 화면에 반영되어야 한다.

### NFR-PERF-002 동시 접속

MVP 기준으로 다음 부하를 지원하는 것을 목표로 한다.

* 동시 사용자 50명
* 사용자당 평균 활성 탭 3개
* 총 활성 Runspace 150개

실제 허용 수치는 서버 CPU, 메모리 및 PowerShell 모듈 사용량을 기준으로 부하 테스트 후 결정한다.

### NFR-PERF-003 출력 제한

대량 출력으로 서버와 브라우저가 과부하되지 않도록 다음 제한을 적용할 수 있어야 한다.

* SignalR 메시지 최대 크기
* 명령당 최대 출력 크기
* 초당 출력 메시지 수
* xterm.js 스크롤백 최대 줄 수
* 명령 실행 최대 시간

---

## 10.2 가용성

### NFR-AVL-001 장애 격리

한 사용자의 PowerShell 실행 오류가 다른 사용자의 세션이나 전체 웹 애플리케이션에 영향을 주지 않아야 한다.

### NFR-AVL-002 세션 비영속성

PowerShell 세션은 애플리케이션 메모리에 유지되며 서버 재시작 시 복구되지 않는다.

서버 재시작 후 사용자는 새 탭을 생성해야 한다.

### NFR-AVL-003 상태 확인

운영 환경은 애플리케이션 및 데이터베이스 연결 상태를 확인할 수 있는 Health Check 엔드포인트를 제공해야 한다.

Health Check에는 민감한 시스템 정보를 포함하지 않는다.

---

## 10.3 보안

### NFR-SEC-001 전송 보안

운영 환경의 모든 HTTP 및 SignalR 통신은 HTTPS를 사용해야 한다.

### NFR-SEC-002 권한 격리

애플리케이션 프로세스는 전용 저권한 서비스 계정으로 실행해야 한다.

### NFR-SEC-003 PowerShell 세션 제한

기본 PowerShell 세션 전체를 그대로 노출하지 않고 업무 목적에 맞는 제한된 실행 환경 적용을 우선 검토한다.

가능한 통제 방법은 다음과 같다.

* 허용 명령어 목록
* 금지 명령어 목록
* 허용 모듈 제한
* Language Mode 제한
* JEA 또는 별도 엔드포인트 사용
* 파일 시스템 접근 경로 제한
* 외부 프로세스 실행 제한
* 네트워크 접근 제한

단순 문자열 기반 금지어 필터만으로는 우회 가능성이 높으므로 핵심 보안 통제로 사용해서는 안 된다.

### NFR-SEC-004 CSRF 방어

쿠키 인증을 사용하는 상태 변경 API에는 CSRF 방어를 적용해야 한다.

SignalR Hub의 Origin 검증과 CORS 정책도 제한적으로 구성한다.

### NFR-SEC-005 XSS 방어

PowerShell 출력은 신뢰할 수 없는 문자열로 취급해야 한다.

출력 내용을 HTML로 직접 삽입하지 않고 xterm.js의 안전한 출력 API를 사용해야 한다.

### NFR-SEC-006 세션 소유권 검증

모든 탭 생성, 명령 실행, 중단 및 종료 요청에서 현재 연결과 인증 사용자가 해당 세션의 소유자인지 검증해야 한다.

### NFR-SEC-007 비밀 정보 관리

데이터베이스 연결 문자열, 쿠키 서명 키 및 기타 비밀값은 소스 코드나 저장소에 포함하지 않는다.

환경 변수, Secret Manager 또는 운영 비밀 관리 시스템을 사용한다.

### NFR-SEC-008 보안 헤더

운영 환경에서는 다음 보안 헤더 적용을 검토한다.

* Content-Security-Policy
* X-Content-Type-Options
* Referrer-Policy
* Permissions-Policy
* Strict-Transport-Security
* 적절한 Frame Ancestors 정책

## 14. 상세 보안 정책

## 14.1 PowerShell 실행 정책

WebPowerShell은 사실상 서버 측 원격 명령 실행 기능을 제공하므로 일반 웹 애플리케이션보다 높은 보안 수준이 필요하다.

다음 중 하나 이상의 방식으로 실행 범위를 제한해야 한다.

1. 승인된 명령어만 실행하는 Allowlist 방식
2. 승인된 PowerShell 함수만 제공하는 제한 세션
3. JEA 기반 역할별 명령 제한
4. 컨테이너 또는 격리된 Worker 프로세스에서 실행
5. 사용자별 또는 세션별 별도 프로세스 격리
6. 네트워크 및 파일 시스템 권한 제한

`InitialSessionState.CreateDefault()`는 기본 명령을 광범위하게 노출할 수 있으므로 운영 환경의 최종 보안 모델로 간주해서는 안 된다.

## 14.2 명령어 검증

문자열에서 특정 단어를 차단하는 방식만으로는 다음과 같은 우회를 막기 어렵다.

* 별칭 사용
* 문자열 조합
* Invoke-Expression
* Reflection
* 외부 프로세스 실행
* 인코딩된 명령
* 모듈 또는 스크립트 로딩

따라서 명령어 필터링보다 실행 권한과 세션 구성을 제한하는 방식을 우선 적용한다.

## 14.3 네트워크 접근

PowerShell 세션에서 외부 인터넷 또는 내부 중요 시스템으로 접근할 필요가 없다면 운영체제 방화벽과 네트워크 정책으로 차단해야 한다.

## 14.4 파일 시스템 접근

전용 서비스 계정은 업무에 필요한 디렉터리에만 접근할 수 있어야 한다.

다음 경로에 대한 쓰기 권한은 최소화한다.

* 운영체제 시스템 디렉터리
* 애플리케이션 배포 디렉터리
* 다른 서비스의 설정 디렉터리
* 인증서 저장소
* 사용자 프로필
* 비밀값 저장 위치

## 15. 기술 제약 및 설계 유의사항

### 15.1 xterm.js와 PowerShell SDK의 차이

xterm.js는 터미널 화면을 렌더링하지만 `System.Management.Automation`의 Runspace는 운영체제의 실제 PTY 또는 ConPTY와 동일하지 않다.

따라서 MVP는 다음 모델을 사용한다.

* 클라이언트가 완성된 명령 문자열을 서버로 전송
* 서버가 PowerShell SDK로 명령 실행
* PowerShell Stream을 SignalR로 전송
* 클라이언트가 프롬프트를 직접 표시

다음 기능은 추가 설계 없이는 완전하게 지원되지 않을 수 있다.

* 대화형 입력
* 전체 화면 콘솔 프로그램
* 방향키 기반 명령 기록
* 탭 자동 완성
* Ctrl+C와 콘솔 시그널의 완전한 호환
* 호스트 UI를 직접 사용하는 Cmdlet
* 네이티브 콘솔 애플리케이션의 화면 제어

### 15.2 비동기 출력 콜백

출력 콜백은 비동기 SignalR 전송을 안전하게 기다릴 수 있는 구조여야 한다.

`Action<string>`에 `async` 람다를 전달하면 `async void` 형태가 되어 예외 추적과 전송 완료 보장이 어려울 수 있다.

인터페이스는 다음과 같은 비동기 콜백 형태를 권장한다.

```csharp
Func<string, CancellationToken, Task>
```

### 15.3 DbContext 수명

SignalR Hub 메서드는 장시간 실행될 수 있으므로 동일한 DbContext를 명령 전체 실행 시간 동안 불필요하게 유지하지 않아야 한다.

감사 로그 저장은 `IDbContextFactory<AppDbContext>` 또는 별도 감사 로그 서비스를 통해 짧은 범위로 수행하는 것을 권장한다.

### 15.4 세션 동시성

`PowerShell` 인스턴스는 동일 세션에서 임의로 병렬 호출해서는 안 된다.

세션별 `SemaphoreSlim`, Channel 기반 명령 큐 또는 전용 실행 루프를 사용해야 한다.

### 15.5 서버 확장

PowerShell 세션이 서버 메모리에 저장되므로 여러 WebAPI 인스턴스를 운영할 경우 Sticky Session이 필요하다.

장기적으로는 다음 구조를 검토한다.

* WebAPI와 PowerShell Worker 분리
* 세션별 독립 Worker 프로세스
* 메시지 브로커 기반 명령 전달
* SignalR Scale-out
* Worker 상태 및 세션 라우팅 저장소

Redis만으로는 실제 Runspace 객체를 공유할 수 없으므로 세션 라우팅 용도로만 사용해야 한다.

## 18. MVP 수용 기준

다음 조건을 모두 충족하면 MVP 출시 가능 상태로 판단한다.

1. 인증되지 않은 사용자는 터미널 Hub에 접근할 수 없다.
2. 유효한 사용자는 로그인 후 터미널 화면에 접근할 수 있다.
3. 비밀번호 변경 후 168시간이 경과한 사용자는 강제로 비밀번호를 변경해야 한다.
4. 사용자는 최소 3개 이상의 독립 탭을 생성하고 사용할 수 있다.
5. 각 탭의 현재 경로와 PowerShell 변수 상태가 독립적으로 유지된다.
6. 탭 전환 후 기존 출력이 손실되지 않는다.
7. 명령 출력과 오류가 해당 탭에만 표시된다.
8. 사용자가 실행한 모든 명령 시도가 감사 로그에 기록된다.
9. 다른 사용자의 세션에 접근할 수 없다.
10. 탭 종료 시 Runspace와 PowerShell 객체가 해제된다.
11. 브라우저 연결 종료 시 해당 연결의 세션이 모두 정리된다.
12. 30분 이상 유휴 상태인 세션이 자동 종료된다.
13. 동일 탭에서 동시 명령 실행으로 인한 PowerShell 객체 충돌이 발생하지 않는다.
14. 로그인 Rate Limiting이 정상 작동한다.
15. 운영 애플리케이션이 전용 저권한 서비스 계정으로 실행된다.
16. 주요 보안 및 동시성 테스트를 통과한다.
17. 부하 테스트에서 정한 동시 사용자 및 Runspace 목표를 충족한다.
18. 세션 반복 생성과 종료 후 지속적인 메모리 증가가 발생하지 않는다.

## 22. 오픈 이슈 및 의사결정 필요 항목

1. 사용자가 모든 PowerShell 명령을 실행할 수 있는지, 승인된 명령만 실행할 수 있는지 결정이 필요하다.
2. 대상 서버가 WebPowerShell 호스트 한 대인지 원격 서버까지 포함하는지 결정이 필요하다.
3. 대화형 PowerShell 명령 지원 범위를 확정해야 한다.
4. 명령어 원문을 감사 로그에 저장할 때 민감 정보 처리 정책이 필요하다.
5. 최대 탭 수, 명령 실행 제한 시간, 최대 출력 크기의 운영 기준을 확정해야 한다.
6. SQL Server와 PostgreSQL 중 운영 표준 데이터베이스를 선정해야 한다.
7. 사용자 계정 발급 및 비활성화 절차를 정의해야 한다.
8. 감사 로그 조회 권한과 보존 기간을 결정해야 한다.
9. 비밀번호 7일 만료 정책이 조직 보안 기준에 부합하는지 검토해야 한다.
10. 고가용성 또는 다중 인스턴스 배포가 MVP에 필요한지 결정해야 한다.
11. PowerShell 실행을 WebAPI 프로세스 내부에서 수행할지 별도 Worker로 격리할지 결정해야 한다.
12. 명령 실행 결과와 감사 로그 저장 순서를 결정해야 한다. 명령 시도 로그를 먼저 저장한 후 결과 상태를 갱신하는 방식을 권장한다.

## 23. 최종 제품 정의

WebPowerShell MVP는 인증된 운영 사용자가 웹 브라우저에서 여러 개의 독립된 PowerShell 세션을 생성하고, 명령어를 실행하며, 결과를 실시간으로 확인할 수 있는 보안 중심의 웹 터미널 시스템이다.

제품은 단순히 PowerShell 화면을 웹으로 옮기는 것에 그치지 않고 다음을 핵심 가치로 한다.

* 사용자별 접근 통제
* 탭별 실행 환경 격리
* 모든 명령 실행의 감사 가능성
* 유휴 세션과 연결 종료에 대한 확실한 자원 회수
* 서버 최소 권한 원칙
* 운영 환경에서 통제 가능한 명령 실행 범위

운영 배포 전에는 PowerShell 전체 권한 노출 여부를 반드시 검토해야 하며, 가능하면 제한된 명령 세션 또는 별도 실행 Worker를 적용해야 한다.
