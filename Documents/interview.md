# TeruTeruShell Architecture Interview (No PTY)

이 문서는 PTY 기반 가상 터미널 환경의 고질적인 교착(Deadlock) 및 외부 콘솔(agy 등) 풀 버퍼링 문제를 해결하기 위해, 주인님(/grill-me 인터뷰)과 합의한 차세대 백엔드 쉘 엔진인 **TeruTeruShell**의 핵심 요구사항을 기록한 것입니다.

## 1. 아키텍처 및 백엔드 엔진
*   **결정 사항:** 기존 `powershell.exe`를 파이프 리다이렉션으로 강제 구동하던 가짜 PTY 래퍼를 전면 폐기하고, C# 기반으로 직접 구현한 **전면 독자 쉘(TeruTeruShell)**을 백엔드 엔진으로 채택합니다.
*   **이유:** 외부 터미널 의존성 및 출력 버퍼링(Full Buffering) 현상을 원천 차단하고 통신 데이터를 완벽하게 통제하기 위함입니다.

## 2. 명령어 스펙 (Built-ins Scope)
*   **결정 사항:** 단순한 우회 실행기(Pass-through)가 아닌, 포괄적인 쉘 기능을 내장합니다.
*   **포함 범위:** 
    *   기본 상태 및 파일 관리: `cd`, `dir`, `ls`, `mkdir`, `rm`, `cp` 등.
    *   기초 유틸리티 및 특수 명령: `ping`, 안티그래비티 전용 도구 `agy` 호출 기능 등.

## 3. 샌드박싱 및 권한 제어 (Directory Isolation)
*   **결정 사항:** 사용자 계정 등급에 따라 파일 시스템 접근 권한(작업 디렉토리)을 엄격히 분리합니다.
*   **관리자 (Admin):** 윈도우 OS의 전체 파일 시스템(`C:\`, `D:\` 등)에 대한 자유로운 이동 및 조작 권한을 부여합니다.
*   **일반 사용자 (User):** 백엔드 서버 실행 파일 경로 하위에 `home/{username}/` 디렉토리를 자동 생성하고, 해당 사용자 세션은 오직 그 내부에서만 이동 및 조작이 가능하도록 제한(Chroot 격리)합니다. 상위 디렉토리(`..`) 접근 시도를 차단해야 합니다.

## 4. 통신 프로토콜 (Data Formatting)
*   **결정 사항:** 기존의 Raw String 기반 스트리밍을 버리고, **구조화된 JSON 객체**를 SignalR로 전송합니다.
*   **데이터 구조 예시:** 
    ```json
    { 
      "type": "stdout", 
      "text": "명령어 실행 결과", 
      "color": "green",
      "timestamp": "2026-06-30T..."
    }
    ```
*   **출력 전달 방식:** **옵저버 패턴(Observer Pattern)을 통한 동기 스트리밍**. 외부 프로그램(또는 내장 명령) 구동 중 출력이 발생할 때마다 옵저버 이벤트를 발생시켜, 버퍼링 대기 없이 즉각 프론트엔드로 JSON 패킷을 방출합니다.

## 5. 프론트엔드 UI 렌더링
*   **결정 사항:** HTML DOM 커스텀 렌더러로 교체하지 않고, 기존 **xterm.js** 컴포넌트를 유지합니다.
*   **동작 방식:** 프론트엔드(`app.js`)는 SignalR을 통해 수신된 JSON 패킷을 파싱하고, `type`이나 `color` 속성에 맞춰 적절한 **ANSI Escape Code**로 변환한 뒤 xterm.js 화면에 그려냅니다. 
*   이를 통해 백엔드는 순수 데이터(JSON)만 다루고, 화면 꾸미기는 프론트엔드가 전담하는 완벽한 역할 분리(Separation of Concerns)를 달성합니다.
